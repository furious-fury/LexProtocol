# Phase 5 Security Hardening Plan

Phase 5 is a security-first pass across the contracts, backend services, indexer, tests, and operational docs. It intentionally does not include frontend work. The goal is to make the protocol safer before any user-facing UI depends on it.

## Objectives

- Harden the smart contracts against reentrancy, bad configuration, accounting drift, replay attacks, and unsafe admin operations.
- Harden the pricing service so settlement signatures cannot be replayed, duplicated, or generated by unauthenticated public callers.
- Harden the indexer so it only trusts events from the configured factory and markets created by that factory.
- Add fuzz, invariant, negative-path, and security regression tests.
- Document the security model, trust assumptions, and deployment checklist for Monad.

## Current High-Priority Findings

### 1. Reentrancy surfaces

`MarketContract` and `Vault` should use `ReentrancyGuard`.

Relevant flows:

- `buyYes()`
- `buyNo()`
- `redeem()`
- `redeem(uint256)`
- `submitOutcome()`
- `Vault.redeemPayout()`
- `Vault.withdrawUnreserved()`

Why this matters:

- ERC1155 minting can call receiver hooks when the buyer is a contract.
- Vault redemption sends native MON to users.
- Even if current accounting mostly follows checks-effects-interactions, a guard gives stronger protection and clearer intent.

### 2. In-memory pricing nonce store

The pricing service currently uses an in-memory global nonce store.

This is acceptable for local development, but not for public testnet or production because process restarts can lose nonce history.

Phase 5 should replace it with a persistent Postgres-backed nonce/signature store.

### 3. Indexer event spoofing risk

The indexer should not accept market lifecycle events from arbitrary contracts that emit matching event topics.

Rules to enforce:

- `MarketCreated` must only be accepted from the configured `MarketFactory`.
- Market events must only be accepted from market addresses previously created by that factory.
- Unknown emitters should be ignored and logged.

### 4. Missing zero-address checks

Admin setters and constructors should reject zero addresses consistently.

Contracts to review:

- `MarketFactory`
- `MarketContract`
- `Vault`
- `OracleRegistry`
- `ResolutionEngine`
- `PositionToken`

### 5. No max position or market exposure caps

The protocol should limit risk per user and per market.

Recommended MVP controls:

- `maxPositionPerUser`
- `maxTotalMarketExposure`
- optional side-specific caps for YES and NO exposure

These should be configurable through factory-level defaults and passed into each new market.

### 6. Market finalization needs stronger accounting

Markets currently resolve and support redemption, but Phase 5 should add explicit finalized-state accounting.

Recommended behavior:

- Track total winning supply.
- Track redeemed winning amount.
- Mark the market `FINALIZED` once all winning collateral has been redeemed.
- Emit `MarketFinalized(uint256 indexed marketId)`.

## Contract Hardening Tasks

### Reentrancy protection

- Add OpenZeppelin `ReentrancyGuard` where needed.
- Mark buy, redeem, resolution, payout, and owner withdrawal flows `nonReentrant`.
- Add regression tests with a malicious ERC1155 receiver or payout receiver where practical.

### Address validation

Add a shared pattern for zero-address validation.

Recommended error:

```solidity
error ZeroAddress();
```

Apply checks to:

- constructor dependency addresses
- oracle address updates
- factory address updates
- market authorization
- token and vault dependency wiring

### Exposure caps

Add caps for:

- per-user minted exposure
- total market exposure
- optionally per-side market exposure

Important design note:

Because ERC1155 positions are transferable, a per-user cap should be enforced on direct purchases/minted exposure, not on current token balance. Enforcing current balance caps would conflict with secondary transfers.

### Vault accounting

Add or test these invariants:

- reserved collateral never exceeds vault balance
- owner cannot withdraw reserved collateral
- losing-side collateral is released after settlement
- winning-side collateral cannot be overpaid
- double redemption is impossible
- failed payout reverts cleanly

Recommended views:

```solidity
function availableBalance() external view returns (uint256);
function reservedBalance() external view returns (uint256);
```

### Oracle validation

Add tests and hardening for:

- zero oracle address
- wrong chain ID
- wrong verifying contract
- wrong market ID
- expired submission
- reused nonce
- malformed signature
- high-S signature rejection
- invalid outcome

Keep global used-nonce replay protection unless a stronger monotonic nonce policy is deliberately chosen.

### Ownership and admin safety

Consider replacing `Ownable` with `Ownable2Step` for:

- `MarketFactory`
- `Vault`
- `OracleRegistry`
- `ResolutionEngine`
- `PositionToken`

This reduces the risk of transferring ownership to the wrong address.

### Emergency controls

Recommended minimal approach:

- pause market creation
- pause trading on new/open markets
- do not pause user redemption unless a critical vault issue is detected

This protects users from admin misuse while still giving the protocol a way to stop new exposure.

## Pricing Service Hardening Tasks

### Persistent nonce store

Replace in-memory nonce generation with Postgres.

Recommended table:

```sql
CREATE TABLE pricing_nonces (
    nonce BIGINT PRIMARY KEY,
    market_id NUMERIC NOT NULL,
    outcome SMALLINT NOT NULL,
    expiry BIGINT NOT NULL,
    signature TEXT NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Requirements:

- nonce allocation must be atomic
- duplicate nonce insertion must fail
- signed payload should be stored for auditability
- restart should not reuse old nonces

### API protection

Keep informational price reads public, but protect signature generation.

Recommended environment variable:

```bash
export PRICING_API_TOKEN="replace-with-a-strong-token"
```

Protected endpoint behavior:

- require `Authorization: Bearer <token>`
- return `401` for missing/invalid token
- avoid logging token values

### Runtime safety

Add:

- request timeouts
- graceful shutdown
- structured logs
- private key redaction
- startup logging for signer address, chain ID, and oracle registry address only

## Indexer Hardening Tasks

### Trusted emitter filtering

Implement known-market filtering.

Flow:

1. Accept `MarketCreated` only from configured factory.
2. Store created market address.
3. Accept market events only from stored market addresses.
4. Ignore unknown emitters.

### Reorg hardening

Improve reorg handling by:

- persisting block parent hashes
- verifying canonical parent continuity
- rolling back to the last known common ancestor
- re-promoting confirmed events after rollback

### Backfill safety

Add chunked backfill.

Recommended config:

```bash
export INDEXER_START_BLOCK=0
export INDEXER_BACKFILL_CHUNK_SIZE=2000
export INDEXER_CONFIRMATIONS=2
```

This keeps RPC calls bounded and safer on public Monad RPC endpoints.

### SSE hardening

Add:

- heartbeat events
- client connection limits
- write deadlines
- clear CORS policy before public deployment

## Test Plan

### Foundry unit tests

Add tests for:

- zero-address constructor/setter failures
- unauthorized admin calls
- max user position limit
- max market exposure limit
- buy after lock failure
- duplicate resolution failure
- redemption before resolution failure
- losing token redemption failure
- partial redemption accounting
- transferred winning token redemption
- owner withdrawal blocked when collateral is reserved

### Foundry fuzz tests

Add fuzz tests for:

- arbitrary buy amounts
- arbitrary redemption amounts
- arbitrary lock times
- arbitrary valid outcomes
- repeated buy/redeem sequences

### Foundry invariant tests

Target invariants:

- `reservedCollateral <= address(vault).balance`
- total redeemed winning collateral never exceeds reserved winning collateral
- market cannot resolve twice
- locked market cannot accept buys
- unauthorized market cannot mint or burn positions

Recommended command:

```bash
cd contracts && forge test --fuzz-runs 10000
```

### Go tests

Add tests for:

- persistent nonce uniqueness
- no nonce reuse after restart
- API token rejection
- valid API token acceptance
- indexer ignores spoofed events
- indexer accepts known factory-created markets
- reorg rollback to common ancestor
- chunked backfill range construction

### Static analysis

Recommended local checks:

```bash
cd contracts && forge fmt
cd contracts && forge build
cd contracts && forge test
cd contracts && forge test --fuzz-runs 10000
go test ./backend/... ./shared/...
go build ./backend/... ./shared/...
```

Optional security scanners:

```bash
slither contracts/src
gosec ./backend/... ./shared/...
```

## Recommended Implementation Order

1. Contract reentrancy guards, zero-address checks, and ownership safety.
2. Exposure caps and finalized-state accounting.
3. Vault accounting views and invariant tests.
4. Persistent pricing nonce store and signed endpoint auth.
5. Indexer trusted-emitter filtering and backfill chunking.
6. Reorg hardening.
7. Fuzz, invariant, static-analysis, and docs pass.

## Phase 5 Completion Criteria

- Contracts pass unit, fuzz, and invariant tests.
- Vault collateral invariants are covered.
- Oracle replay and expiry behavior are covered.
- Pricing service does not reuse nonces across restarts.
- Signature-generation endpoint is protected.
- Indexer ignores spoofed events from unknown emitters.
- Backfill is chunked and configurable.
- Reorg handling has regression tests.
- Security model and deployment checklist are documented.
